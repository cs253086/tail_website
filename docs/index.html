TAIL OS documentation

TAIL OS is a microkernel real-time operating system written in Rust. It's built for robotics, autonomous vehicles, and other safety-critical systems where timing is a hard requirement, not a nice-to-have.

Version: v0.9.0 · Early preview

Overview

v0.9.0 ships with:

  • Microkernel — only process, memory, scheduling, and IPC run privileged. Drivers and everything else are userspace.
  • Real-time scheduler — 64 priority levels, preemptive, O(1) dispatch, FIFO or Round-Robin policy.
  • Two IPC patterns — synchronous request/reply (Service) and asynchronous publish/subscribe (Topic), both kernel-mediated with read-only delivery.
  • Userspace drivers — UART and SD card (SDHC) on Raspberry Pi 3B.
  • Developer shell (tsh>) — inspect tasks, processes, memory, and drivers.

Microkernel

The kernel keeps a small, fixed set of services in privileged mode and pushes everything else out to userspace. Four subsystems sit above a hardware abstraction layer; user applications reach them through a single kernel call interface.

Core services

  • Process management — create and terminate processes, isolate each one in its own address space, assign a unique PID.
  • Memory management — allocate and protect memory, give each process a private virtual address space, support shared memory regions when processes opt in.
  • Thread scheduling — pick the next thread to run on each CPU according to priority.
  • IPC — route messages and service calls between processes.

Processes and threads

A process owns an isolated address space, a PID, and resource limits. A thread is a unit of execution inside a process — threads in the same process share memory but have their own stack and execution context. Concurrent execution lives at the thread level; isolation lives at the process level.

Memory model

Memory is split into kernel space (privileged), user space (per-process, isolated), and explicitly created shared regions. The kernel enforces read / write / execute permissions through virtual memory; an invalid access faults rather than corrupting another process.

Targets

AArch64 and x86_64. Written in Rust, so memory-safety errors that plague C kernels are eliminated at compile time.

Scheduler

The scheduler is priority-based and preemptive. Its decision rule is simple: always run the highest-priority thread that is ready. What makes it suitable for real-time work is that every scheduling step is bounded in time, regardless of how many threads exist.

Policy

  • 64 priority levels (0 = idle, 63 = highest).
  • Highest-priority-first with FIFO ordering within a priority — threads run in the order they became ready.
  • FIFO or Round-Robin — system-wide policy. The default is Round-Robin with a 10 ms quantum; switching to FIFO removes time slicing entirely.
  • O(1) dispatch — the next thread is found with a single hardware count-leading-zeros instruction over a 64-bit ready bitmap, so dispatch latency is constant.

Thread states

A thread is always in exactly one of four states: Ready (waiting for the CPU), Running (executing now), Blocked (waiting on IPC, I/O, a mutex, sleep, or a timeout), or Terminated.

Real-time properties

  • Bounded ready-to-running latency — constant-time dispatch and a fully preemptible kernel keep the worst case computable.
  • Priority inheritance — when a high-priority thread waits on a mutex held by a low-priority one, the holder temporarily inherits the higher priority. Same boost is applied automatically across synchronous IPC calls. This prevents classical priority inversion.
  • Preemptible kernel — syscalls themselves can be preempted, so a long kernel operation doesn't hold off a higher-priority thread. Interrupts stay enabled across syscall execution.
  • SMP — a single global ready queue is shared across CPUs; threads honour a per-thread CPU affinity mask. Only CPU 0 takes the periodic timer tick; secondaries are nudged by reschedule IPIs.
  • Sleep & timeout queues — time-based blocking is sorted by wake deadline; expired entries are woken on every tick.

IPC

TAIL OS provides two complementary IPC mechanisms: Service IPC for synchronous request/reply and Topic IPC for asynchronous publish/subscribe. Both share one guarantee: the kernel is the sole mediator. Processes never write into each other's memory — every payload is copied once into a kernel-owned buffer and exposed to receivers as a read-only mapping. That single rule is what makes the IPC layer analysable for safety-critical use (designed with ISO 26262 and IEC 61508 in mind).

Shared message pool

Both mechanisms draw from a single tiered pool of kernel-owned buffers — 256 B, 4 KiB, and 64 KiB cells by default, sized for typical message payloads. In certified mode the pool is pre-allocated at boot and never calls into the heap on the IPC hot path, giving you fixed, predictable resource use. A critical service can also reserve guaranteed buffer slots so it never starves under load.

Service IPC — synchronous request / reply

A service is a named endpoint owned by a server thread. A client sends a request and blocks until the reply arrives; the kernel routes the call to the registered server, copies the request into a pool cell, and hands the server a read-only view. Use it for device drivers, system services, and anything RPC-shaped.

  • Bounded queues. Each service has a 16-slot request queue; each process can have at most 8 requests in flight at once.
  • Priority inheritance. While the server handles a request, it runs at the client's priority — a low-priority server cannot hold off a high-priority client.
  • Per-request timeouts. Clients can attach a deadline to any send and get a clean TimedOut error instead of waiting forever.
  • Clean teardown. If the server thread or process dies, the kernel drains its queue, releases the buffers, and unblocks every waiting client.

Topic IPC — asynchronous publish / subscribe

A topic is a named channel. Any number of publishers send messages to it; every subscriber that was subscribed at publish time gets its own read-only view of the same kernel buffer. Use it for sensor streams, event broadcasting, and data logging.

  • Fire-and-forget. The publisher never blocks on subscribers. It returns as soon as the kernel has staged the message.
  • One copy, many readers. The kernel copies the payload once and refcounts the pool cell — fan-out to N subscribers stays cheap.
  • Drop-oldest queues. Each subscriber has an 8-slot pending ring. A slow subscriber loses its oldest unread message rather than slowing the publisher or the other subscribers.
  • Per-subscriber isolation. A crashed or stuck subscriber can't stall anyone else; the kernel cleans up its buffers on process death.

Picking between them

Reach for Service IPC when you need a reply, a guarantee of delivery, or transactional semantics — driver calls, configuration changes, filesystem operations. Reach for Topic IPC when you have a continuous stream of data and the latest sample matters more than every sample — IMU readings, camera frames, status broadcasts.

FAQ

Is TAIL OS open source?

No — TAIL OS is currently a closed-source preview. Early access is shared for evaluation under agreement; contact us if you'd like to take it for a spin.

Which boards are supported?

Raspberry Pi 3B and QEMU virt (aarch64) today. Cortex-M and RISC-V targets are in progress.

Why a microkernel?

Smaller attack surface, drivers can crash without taking the kernel down, and it's far easier to reason about determinism when the kernel only does three things.
